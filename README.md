## Implementation Details

## Memory/Storage for VibeMeter
Where the data about total line counts and ownership of lines for each project is stored.

All data is stored in `~/.vibemete/` directory:
```
~/.vibemeter/
├── projects.json
├── projects/
│   └── <project-id>/
│       ├── events.jsonl
│       └── ownership.json
└── tmp/
    ├── <session>-<filehash>.txt
    └── <filehash>.expected
```

Explanation of above structure:
- Every time you run `vibemeter init` in a new project, a new entry is stored in `projects.json` of the shape `{ "project-id": { path, name, initAt } }`.
- For every project initialized with vibemeter, a project data folder is created in `./vibemeter/projects` with a unique ID so two projects can't collide.
- `events.jsonl` is the source of truth. It's an append-only log, a single JSON event per line.
- `ownership.json` is a line level map. For each tracked file, an array where index i = who wrote line i ("claude", "human","starter","scaffold"). 
  - This is needed because `events.jsonl` alone can't attribute deletions. When a human deletes 10 lines, we need to know whose lines those were to decrement the right bucket. The ownership map answers that.
- `tmp/` is a short-lived scratch space for the hook <-> tracker handshake. The contents in this directory are disposable.
- `<session>-<filehash>.txt` is the pre-Write snapshot written by the pre-hook before Claude edits a file.
  - The post-hook diffs the current file against this snapshot to compute exactly how many lines Claude added/removed. 
  - Keyed by session + file hash so concurrent edits don't clobber each other.
- `<filehash>.expected` --> this file is used for the vscode extension, so I will come to it in a bit.



## Project Structure
What scripts in the project source code do and the factorization (directory structure) of the project codebase explained.

```
VibeMeter/
├── packages/
│   ├── core/
│   ├── cli/
│   ├── hooks/
│   └── vscode/
├── build.js
├── pnpm-workspace.yaml
├── package.json
├── vitest.config.ts
└── tsconfig.base.json
```

### Top-level files
- `build.js`
- `pnpm-workspace.yaml` tells pnpm that this is a monorepo. The real projects live under `packages/*`. Without this pnpm treats the root as a single project. The 4 packages need to treated separately as they are for different usecases.
- `package.json` (root) contains shared dev tooling: vitest, esbuild, typescript, plus repo-wide scripts like `pnpm build`, `pnpm test` that fan out to each of the packages.
- `vitest.config.ts` Vitest runs your tests from tests/ (organized by unit/ integration/ e2e/ stress/ per your project rules). This file tells Vitest how to run them:
    - Which files are tests — usually a include glob like `tests/**/*.test.ts`.
    - Test environment — `node` (for core/cli/hooks) vs `jsdom` (for the VS Code React webview, which needs a fake DOM).
    - Coverage settings — since you're using `@vitest/coverage-v8`, this is where you set the provider, which files count toward coverage, and thresholds (e.g. "fail if `<80%` covered").
    - Path aliases / workspace setup — in a monorepo like this, it tells Vitest how to resolve imports across `packages/core`, `packages/cli`, etc., so a test in `tests/` can `import { appendEvent } from "@VibeMeter/core`" without hacks.
    - Globals, setup files, timeouts — e.g. a `setup.ts` that creates a temp `~/.VibeMeter/` before each test and cleans up after.
    - Your tests live in a single top-level `tests/` folder that spans all four packages. One root config runs them all with one command (`pnpm test`), produces one coverage report, and avoids duplicating settings across `packages/*/vitest.config.ts`.
    - It's the knobs-and-dials file that makes `pnpm test` actually know what to do.
- `tsconfig.base.json` the TypeScript settings every package extends. Further each of the packages has its own `tsconfig.json` that just says `"extends": "../../tsconfig.base.json"` and adds its own `include`/`outDir`.

### The 4 packages
The split exists because each package has a different runtime: the hooks run as standalone Node scripts spawned by Claude Code, the VS Code extension runs inside VS Code's extension host, the CLI runs in your terminal. They can't be one package — but they all need the same types and log format, which is why core exists.

1. `packages/core/` — the shared library
- Everyone else depends on this. No entry point of its own; it's a library.
    - `src/types.ts` — every shared TypeScript type: `Event`, `Source` (`"claude" | "human" | "starter" | "scaffold"`), `Project`, `Stats`. One file so there's no ambiguity about where a type lives.
    - `src/log.ts` — the `.jsonl` event log API: `appendEvent()`, `readEvents()`, `computeStats()`. This is the only code that knows the on-disk log format — hooks, CLI, and extension all go through it. Change the format here, everyone updates.
    - `src/ownership.ts` — reads/writes `ownership.json` (the per-line "who wrote this line" map). Exposes helpers like `getOwner(file, line)` and `updateOwnership(file, diff, source)`. Needed so deletions can be attributed to the right bucket.
    - `src/projects.ts` — reads/writes `projects.json`, the global project registry. `registerProject()`, `findProjectByPath()`, `listProjects()`.

2. `packages/cli/` — the VibeMeter terminal command
    - `src/index.ts` — commander setup: wires subcommands, parses argv, dispatches.
    - `src/commands/init.ts` — the heavy one: snapshot the baseline (all lines → starter), detect scaffolding tools (mark those lines → scaffold), install the Claude Code hooks into the project's settings, register the project in projects.json.
    - `src/commands/stats.ts` — calls core/log.computeStats(), prints a table.
    - `src/commands/export.ts` — same stats, but to JSON/CSV for external tools.
    - `src/commands/serve.ts` — tiny local HTTP server that renders the web dashboard (same UI as the VS Code webview, just in a browser). For people who don't use VS Code.
    - `src/commands/projects.ts` — lists everything in projects.json.

3. `packages/hooks/` — the Claude Code hook scripts
- Claude Code has a hook system: before/after certain tool calls (Write, Edit, MultiEdit), it spawns scripts you configure. These two files are those scripts.
    - `src/pre-hook.ts` — runs before Claude writes. Reads the target file's current contents and saves them to `~/.VibeMeter/tmp/<session>-<filehash>`.txt. This is the "before" snapshot.
    - `src/post-hook.ts` — runs after Claude writes. Loads the pre-snapshot, diffs it against the new file contents, computes added / removed line counts using the diff library, appends a claude event to events.jsonl, updates ownership.json, and drops the <filehash>.expected marker so the VS Code tracker knows "this version of the file is mine."
Two separate files because Claude Code's hook system calls them at two different lifecycle points — there's no way to do both jobs in one script.

4. `packages/vscode/` — the VS Code extension
The user-facing live view. Also the only package with UI code (React + Recharts).
   - `src/tracker.ts` — uses `chokidar` to watch the project's files. On any change: hash the file, check for `<hash>.expected`. If matched → Claude's edit, already logged, ignore. If not → human edit, diff against the last known state, append a human event.
   - `src/statusbar.ts` — the little status bar item at the bottom of VS Code ("🤖 42% Claude"). Subscribes to log changes, recomputes stats, updates the text. [MAY WANT THIS TO BE AT THE TOP]
   - `src/webview.ts` — the sidebar panel. Spins up a VS Code webview, loads the bundled React app, posts stats into it. The React app (Recharts) renders the pie chart / trends.
   - Plus an `extension.ts` entry point (activates the extension, wires the three modules together) — implied by convention though not in the critical-files table.

core knows nothing about the others. The other three never import from each other — only from core. That's why the log format can be trusted: there's exactly one implementation of it.
